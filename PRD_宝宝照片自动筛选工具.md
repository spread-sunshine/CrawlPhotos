## 一、背景与痛点

### 1.1 业务背景
幼儿园老师每天放学后会在QQ群相册中上传当天所有小朋友的活动/学习照片，家长需要从大量照片中手动筛选出自己孩子的照片，费时费力且容易遗漏。

### 1.2 用户痛点

| 序号 | 痛点描述 | 影响程度 |
|-----|---------|---------|
| P1 | 每天需浏览几十至上百张照片逐一筛选，耗时5-15分钟 | 高 |
| P2 | 手动筛选容易遗漏重要瞬间（如孩子第一次上台表演） | 高 |
| P3 | 照片散落在群相册中，无法形成个人成长档案 | 中 |
| P4 | 手机空间有限，无法长期保存所有原始照片 | 中 |
| P5 | 老人/其他家庭成员想看孩子照片但不方便翻群相册 | 低 |

## 二、目标与范围

### 2.1 项目目标
- **核心目标**: 自动化完成"下载→识别→分类→归档→备份"全流程，将家长每日操作时间从5-15分钟降低至0分钟
- **延伸目标**: 构建宝宝成长影像档案系统

### 2.2 MVP功能范围 (V1.0)

| 功能模块 | 功能点 | 优先级 | 说明 |
|---------|-------|-------|------|
| QQ群相册同步 | 自动拉取群相册新增照片 | P0 | 核心功能 |
| 人脸识别引擎 | 基于参考图识别人脸并筛选 | P0 | 核心功能 |
| 本地归档管理 | 按年/月/日目录结构存储 | P0 | 核心功能 |
| 配置管理 | 目标人物、存储路径、QQ账号等配置 | P0 | 基础功能 |
| 日志监控 | 运行日志、异常告警、处理统计 | P1 | 运维保障 |
| 个人相册上传 | 筛选后自动上传到个人QQ相册 | P1 | 增值功能 |

### 2.3 非功能需求

| 维度 | 要求 |
|-----|------|
| **可靠性** | 7x24小时无人值守运行，异常自动恢复 |
| **准确性** | 人脸识别准确率 >=95%，漏检率 <=2% |
| **性能** | 单张照片识别耗时 <500ms，100张批次处理 <60s |
| **安全性** | QQ账号信息安全存储，本地数据不外传 |
| **易用性** | 配置一次即可长期运行，无需日常干预 |

## 三、用户角色与使用场景

### 3.1 用户画像
- **角色名称**: 幼儿园家长(主要使用者)
- **年龄范围**: 25-40岁
- **技术水平**: 普通(会基本电脑操作)
- **核心诉求**: 自动收集宝宝照片，不遗漏任何精彩瞬间

### 3.2 典型使用流程

**Before (无工具):**
打开QQ -> 进入班级群 -> 点击群相册 -> 浏览今日相册 -> 逐张查看是否包含自己孩子 -> 长按保存喜欢的照片 -> 整理到手机相册对应月份文件夹 -> 耗时: 5~15分钟

**After (使用本工具):**
工具后台自动运行 -> 下班后打开目标文件夹查看今日照片 -> 可选: 打开个人QQ相册分享给家人 -> 耗时: 0分钟(纯消费)

## 四、功能详述

### 4.1 QQ群相册数据采集

#### 4.1.1 数据源
- **来源**: QQ群相册(腾讯QQ)
- **触发方式**: 支持三种触发模式（详见4.1.4）
- **采集范围**: 指定QQ群的指定相册

#### 4.1.2 数据模型

Photo 实体包含: photo_id(唯一标识)、album_id、group_id、upload_time、uploader、url、thumbnail_url、file_size、width、height、local_path、processed、contains_target、confidence、created_at、updated_at

#### 4.1.3 同步策略

| 策略项 | 方案 | 说明 |
|-------|------|------|
| 增量同步 | 基于upload_time去重 | 只处理新增照片 |
| 失败重试 | 指数退避,最多重试3次 | 网络抖动容错 |
| 断点续传 | 本地记录已处理photo_id | 重启后不重复处理 |
| 全局去重 | photo_id + file_hash双重校验 | 确保零重复（详见4.6节） |

#### 4.1.4 触发方式设计（三种模式并存）

**模式A: 事件触发（老师上传时自动触发）**
- 监听QQ群消息，检测到"上传了X张照片到相册"类系统通知后立即触发
- 支持方案: QQ机器人协议(go-cqhttp/Lagrange.Core) / 浏览器插件DOM监听 / 高频轮询(伪实时)
- **优势**: 实时性最高；**劣势**: 需保持在线/运行，依赖第三方协议

**模式B: 定时触发（固定时间触发）**
- 使用Cron表达式定义固定时间点执行
- 推荐: 工作日每30分钟(17:00~21:00高频段)，周末每2小时

**模式C: 手动触发（用户主动执行）**
- CLI命令: `python main.py --run [--days N] [--date YYYY-MM-DD]`
- Web API: `POST /api/v1/tasks/trigger`
- 系统托盘图标右键菜单(GUI版本)

### 4.2 人脸识别引擎（可插拔架构）

> **核心设计原则**: 采用**策略模式(Strategy Pattern) + 插件化**架构，
> 所有识别提供商实现统一接口，通过配置文件一行即可切换，零代码改动。
#### 4.2.1 统一接口定义 (IFaceRecognizer)

抽象基类定义以下核心方法:
- `initialize(targets)` - 初始化识别引擎，预加载特征库
- `health_check()` - 健康检查，检测服务可用性
- `detect_faces(image_path)` - 检测图片中的人脸(不含识别/比对)
- `recognize(image_path)` - **核心方法**: 检测+特征提取+1:N搜索一站式调用
- `add_reference_photos(target_name, photo_paths)` - 添加/更新参考照片
- `remove_target(target_name)` - 移除目标人物
- `list_targets()` - 列举已注册的目标人物及状态
- `batch_recognize(image_paths)` - 批量识别(可选优化接口)
- `cleanup()` - 清理资源

配套数据模型: `BoundingBox`(边界框)、`FaceDetection`(单次检测结果)、`RecognitionResult`(完整识别结果)、`TargetConfig`(目标人物配置)、`ProviderInfo`(提供商能力描述)

异常体系: `FaceRecognizerError` -> `ProviderInitError` / `ProviderApiError` / `QuotaExhaustedError` / `ImageInvalidError` / `NoFaceDetectedError` / `TargetNotFoundError`

#### 4.2.2 插件注册与工厂机制

- **FaceRecognizerRegistry**: 单例注册表，维护可用提供商映射，提供 `register()` / `create()` 工厂方法 / `list_available()`
- **FaceRecognizerFacade**: 业务层唯一交互入口，封装提供商选择逻辑(读配置->调工厂创建)、统一异常处理、支持运行时动态切换

#### 4.2.3 内置 Provider 规格

**Provider A: TencentCloudProvider (默认)**
- 准确率最高(99%+)，支持人员库持久化和批量API
- 免费额度1000次/月，需网络连接
- 实现: CreateGroup -> CreatePerson -> DetectFace + SearchFaces

**Provider B: BaiduProvider**
- 免费额度最大(QPS限制内基本免费)，文档清晰
- 准确度略低(约97%)，需网络连接
- 实现: FaceSet管理 + 活体检测选项

**Provider C: InsightFaceLocalProvider (离线方案)**
- 完全离线运行，无限免费，隐私性最好
- CPU模式准确率约95%(vs云端99%)，首次需下载模型(~200MB)
- 硬件要求: 最低4核CPU+8GB内存；推荐GTX 1060以上GPU

**Provider D: FacePlusProvider**
- API友好但商用收费较高，无长期免费额度
- 适用场景: 其他三家都不满足特殊需求时的备选

#### 4.2.4 切换提供商
只需修改 `config.yaml` 中 `face_recognition.provider` 字段值即可切换，重启服务生效，业务代码零改动。

#### 4.2.5 识别流程
输入照片 -> 检测人脸 -> 提取特征向量 -> 与目标人物库比对(1:N搜索) -> 相似度>=阈值则包含目标人物，否则不包含。
### 4.3 本地归档管理

#### 4.3.1 目录结构设计
按年/月/日三级目录归档，每日生成 metadata.json 记录元数据。

#### 4.3.2 文件命名规则
格式: `{YYYYMMDD}_{序号}_{来源标识}.{扩展名}`，示例: `20260416_0001_qqgroup.jpg`

#### 4.3.3 元数据文件 (metadata.json)
每日生成，记录日期、总照片数、目标照片数、来源群组、上传者、各照片的置信度/人脸数/标签等信息。

### 4.4 个人相册上传(可选)

| 策略项 | 方案 | 说明 |
|-------|------|------|
| 上传时机 | 筛选完成后立即上传 | 保证时效性 |
| 相册命名 | `{年份}年{月份}宝宝照片` | 按月自动创建 |
| 去重机制 | 基于照片hash去重 | 避免重复上传 |
| 失败处理 | 加入重试队列 | 网络问题容错 |

### 4.5 配置管理

配置文件 `config.yaml` 包含以下段:
- **qq**: 群相册配置(group_id/album_id/cookies)、个人相册配置
- **face_recognition**: 核心切换开关(provider字段) + 各Provider独立配置段(tencent_cloud/baidu/insight_face_local) + targets目标人物配置
- **storage**: 存储根目录、目录格式、文件名格式
- **scheduler**: Cron表达式、启动扫描、回溯天数
- **logging**: 日志级别、目录、大小限制、保留天数
- **notification**: 通知类型(Webhook/邮件/钉钉)与规则
- 敏感信息使用环境变量引用(`${VAR}`形式)

### 4.6 全局去重机制（核心保障）

#### 4.6.1 三层去重策略
1. **任务入队去重**: 基于photo_id的内存集合(Set)，防多触发源重复入队
2. **数据库持久化去重**: SQLite表 photos_record 记录全生命周期状态，程序重启后仍有效
3. **文件级Hash校验**: SHA256哈希比对，即使photo_id变更也拒绝覆盖

#### 4.6.2 去重流程
- 采集阶段: 查库(source_photo_id) -> 已存在completed/skipped则跳过
- 下载阶段: 计算SHA256 -> 已存在则复用记录
- 存储阶段: 文件存在+size校验 -> 缺记录则补记录不覆盖
- 上传阶段: personal_photo_id检查 -> 已上传则跳过

### 4.7 小程序规划（Phase 5 后续）
> V1.x 使用本地H5/Web界面过渡，V2.0 升级为微信小程序。

## 七、数据处理流程

### 7.1 主处理流程
调度器触发 -> 采集器查询新增照片 -> 逐张下载 -> 人脸识别 -> 判断是否包含目标 -> 归档存储(是)/丢弃(否) -> 发送通知 -> 任务完成

### 7.2 异常处理流程
发生异常 -> 网络超时? -> 指数退避重试(最多3次) -> Cookie失效? -> 告警需重新登录 -> API配额耗尽? -> 告警等待次日重置 -> 最终: 记录错误日志，跳过该照片继续下一张

## 八、运营与监控

### 8.1 关键指标(KPI)

| 指标名称 | 定义 | 目标值 | 监控频率 |
|---------|------|--------|---------|
| 采集成功率 | 成功采集/总应采集 | >=99% | 每日 |
| 识别准确率 | 正确识别/实际包含总数 | >=95% | 每周抽检 |
| 漏检率 | 未识别出/实际包含总数 | <=2% | 每周抽检 |
| 处理及时率 | 2小时内完成占比 | >=95% | 每日 |
| 系统可用性 | 正常运行/总时间 | >=99% | 实时 |

### 8.2 日志规范
- DEBUG: 详细调试信息 / INFO: 正常业务流程 / WARN: 注意事项(Cookie即将过期等) / ERROR: 错误情况
- 格式: `{timestamp} [{level}] [{module}] {message} {extra}`

### 8.3 通知策略
- 发现新照片 -> INFO级微信通知
- 处理完成 -> 每日汇总报告
- Cookie即将过期 -> WARN级预警
- 连续失败/磁盘不足 -> ERROR级告警

## 十、安全与隐私

| 安全维度 | 措施 |
|---------|------|
